Architecture

How It Works

Web2Bridge acts as a Stateless Cryptographic Router with two entropy paths that deterministically generate a wallet from a user's identity.

1

Social Login

User signs in with Google, Apple, GitHub, etc. via your chosen auth provider. The SDK receives a stable, namespaced user ID.

→
2

Entropy Generation

The SDK uses WebAuthn PRF (hardware passkeys) to derive 32 bytes of deterministic entropy — or falls back to a spending password on unsupported devices.

→
3

Wallet Derivation

Entropy is fed through HKDF-SHA-256, converted to a BIP39 24-word mnemonic, and used to create an HD wallet at an app-isolated derivation path.

→
4

Ready to Sign

A CIP-30 compatible Cardano wallet lives in browser memory for signing transactions. Nothing is ever stored or transmitted.

Primary Path

WebAuthn PRF

Hardware-backed passkeys (FaceID, TouchID, YubiKey) generate deterministic entropy via the PRF extension. The private key never leaves the secure enclave.

  • iOS 18+ / Safari 18+
  • Android 14+ / Chrome 130+
  • macOS Sequoia / Chrome 128+
  • Hardware security keys (YubiKey 5, Titan M2)
Fallback Path

Password-Derived

When PRF is unavailable, the user provides a spending password. Entropy is derived via Argon2id (64 MB, 3 iterations) or PBKDF2 — same deterministic result every time.

  • Works on any modern browser
  • Password strength enforced (zxcvbn ≥ 3)
  • Password is never stored
  • Required on every login to re-derive

2

Configure the Provider

Wrap your app with ClerkProvider and Web2BridgeProvider. The SDK accepts an AuthAdapter instance — swap providers by changing one import.

main.tsx
import { ClerkProvider } from "@clerk/clerk-react";
import { Web2BridgeProvider } from "@web2bridge/react";
import { ClerkAdapter } from "@web2bridge/auth-clerk";

// Instantiate the adapter once — pass it to the provider
const authAdapter = new ClerkAdapter({
  publishableKey: "pk_live_...",
});

function App() {
  return (
    <ClerkProvider publishableKey="pk_live_...">
      <Web2BridgeProvider
        adapter={authAdapter}
        config={{
          appDomain: "your-app.com",          // Used to derive app-specific wallet path
          networkId: 0,                       // 0 = Preprod testnet, 1 = Mainnet
          kdf: "hkdf",                        // "hkdf" | "pbkdf2" | "argon2id"
          fallback: {
            enabled: true,                   // Auto-fallback when PRF unavailable
            kdf: "argon2id",                  // "argon2id" | "pbkdf2"
          },
        }}
      >
        <YourApp />
      </Web2BridgeProvider>
    </ClerkProvider>
  );
}
3

Use the Hook

The useWeb2Bridge() hook exposes state and actions. All async methods return Result<T> — never throw.

WalletButton.tsx
import { useWeb2Bridge } from "@web2bridge/react";

function WalletButton() {
  const {
    isAuthenticated,
    wallet,
    entropyPath,       // "prf" | "password" | null
    error,
    login,
    logout,
    exportRecoveryPhrase,
  } = useWeb2Bridge();

  const handleLogin = async () => {
    // For PRF-capable devices, no password needed.
    // For fallback devices, pass the spending password:
    const result = await login({ password: "user-spending-password" });
    if (result.error) {
      console.error(result.error.message);
    }
  };

  if (!isAuthenticated) {
    return <button onClick={handleLogin}>Sign In</button>;
  }

  return (
    <div>
      <p>Entropy path: {entropyPath}</p>
      <button onClick={logout}>Sign Out</button>
    </div>
  );
}
4

Sign Transactions & Data

The wallet object exposes a CIP-30-like interface. All methods return Result<T>.

signing.tsx
const { wallet } = useWeb2Bridge();

// Get wallet addresses
const addrs = await wallet.getUsedAddresses();
// → Result<string[]>

// Sign a transaction (CBOR hex)
const signed = await wallet.signTx(txCborHex, false);
// → Result<string>

// Sign arbitrary data (CIP-8)
const sig = await wallet.signData(address, "Hello Cardano!");
// → Result<string>
5

Export Recovery Phrase

The "escape hatch" — users can export their BIP39 24-word mnemonic at any time. This requires fresh re-verification (PRF re-auth or password re-entry).

RecoveryExport.tsx
const { exportRecoveryPhrase, entropyPath } = useWeb2Bridge();

const handleExport = async () => {
  // PRF path: triggers fresh WebAuthn re-authentication
  // Password path: requires the spending password
  const result = await exportRecoveryPhrase({
    password: entropyPath === "password" ? userPassword : undefined,
  });

  if (result.error) {
    console.error("Verification failed:", result.error.message);
    return;
  }

  // result.data is string[] — the 24-word BIP39 mnemonic
  // Display to user. NEVER log, store, or transmit.
  showMnemonicModal(result.data);
};
Important: The recovery phrase is the user's master key. It can be imported into any standard Cardano wallet (Eternl, Nami, etc.) for full wallet recovery independent of Web2Bridge.

Result Type

All async methods return a Result<T> — a discriminated union that eliminates try/catch boilerplate:

types.ts
type Result<T> =
  | { data: T;    error: null }    // Success
  | { data: null; error: Web2BridgeError }  // Failure

Writing a Custom Adapter

Support any auth provider by implementing the AuthAdapter interface from @web2bridge/core:

MyAdapter.ts
import type { AuthAdapter, Result } from "@web2bridge/core";

export class MyAdapter implements AuthAdapter {
  // Stable identifier — NEVER change after first deploy.
  // This is part of the cryptographic input.
  readonly providerId = "myprovider";

  async login(): Promise<Result<void>> { /* ... */ }
  async logout(): Promise<Result<void>> { /* ... */ }
  async getUserId(): Promise<Result<string>> { /* ... */ }
  isAuthenticated(): boolean { /* ... */ }
}
providerId rules: Must be a lowercase alphanumeric string starting with a letter (e.g. "clerk", "auth0"). Changing it after deployment will silently derive different wallets for all existing users.
Security

Frequently Asked Questions

Questions developers and security engineers ask most about Web2Bridge's architecture.

Where are the private keys stored?

Nowhere. Private keys are never stored — not in localStorage, sessionStorage, IndexedDB, cookies, or any server. The wallet is deterministically re-derived from the user's identity + entropy source on every login. It exists only in browser memory for the duration of the session.

What if your servers get hacked?

There are no servers. Web2Bridge is a client-side SDK with zero backend infrastructure. There is no database to breach, no key vault to compromise, and no API that handles private keys. The auth provider (Clerk, Auth0, etc.) only provides the user's identity — it never sees entropy, keys, or mnemonics.

Can you access my users' wallets?

No. The SDK is open source and runs entirely in the user's browser. Wallet derivation depends on a secret only the user possesses — either a hardware-backed passkey (WebAuthn PRF) or a spending password. Neither the SDK author, the dApp developer, nor the auth provider can reconstruct the wallet without the user's secret.

What happens if the user loses their device?

PRF path: If the passkey is synced via iCloud Keychain or Google Password Manager, the wallet can be recovered on a new device. If not, the user needs their exported recovery phrase.

Password path: The wallet can be re-derived on any device by signing in and entering the same spending password.

Recovery phrase: As a last resort, users can import their 24-word BIP39 mnemonic into any standard Cardano wallet (Eternl, Nami, etc.).

Is the wallet standard or proprietary?

Fully standard. The wallet uses a BIP39 24-word mnemonic, BIP32 HD derivation (m/1852'/1815'/AppID'/0/0), and is CIP-30 compatible. The recovery phrase works with any Cardano wallet that supports BIP39 mnemonic import.

What if the auth provider goes down?

If the auth provider (e.g. Clerk) is temporarily unavailable, users can't initiate a new login. However, since the wallet is derived from the user's identity + secret, the wallet itself is never at risk. Users who have exported their recovery phrase can always import it into a standalone wallet.

Additionally, because the SDK is provider-agnostic, developers can migrate to a different auth provider. As long as the same providerId and userId are preserved, the same wallet will be derived.

Can two different auth providers produce the same wallet?

No. Every user ID is namespaced with the provider identifier before being used as a cryptographic input: clerk:user_abc123 vs auth0:user_abc123. Different providers always produce different wallets, even for the same underlying user.

How strong is the password fallback?

The fallback path uses Argon2id by default (64 MB memory, 3 iterations, parallelism 1) — the current gold standard for password hashing, resistant to GPU and ASIC attacks. Password strength is enforced via zxcvbn with a minimum score of 3. Weak passwords are rejected before derivation begins.

PBKDF2-SHA-256 (210,000 iterations) is available as a lighter alternative where WASM is not feasible.

Is the code audited?

The SDK is open source and available for review on GitHub. The cryptographic primitives rely on the Web Crypto API (browser-native, FIPS-validated implementations) and WebAuthn (hardware-backed). No custom cryptography is invented — the SDK orchestrates standard, well-reviewed building blocks.

What about phishing attacks?

WebAuthn PRF is inherently phishing-resistant — the browser binds the passkey to the origin (domain), so credentials cannot be used on a different site. For the password path, the spending password is combined with the user's namespaced identity, making credential-stuffing attacks ineffective.
